🔒 ACID TRANSACTION IMPLEMENTATION
═══════════════════════════════════════════════════════════════════════════════

BOOKING TRANSACTION FLOW:
───────────────────────────────────────────────────────────────────────────────

START TRANSACTION (SERIALIZABLE ISOLATION)
    ↓
1. VERIFY USER
   ├─ SELECT User WHERE id = user_id FOR UPDATE (Lock)
   └─ If not found: ROLLBACK & Error 404
    ↓
2. VERIFY & LOCK SCHEDULE
   ├─ SELECT Schedule WHERE id = schedule_id FOR UPDATE (Lock)
   ├─ Check status != 'Cancelled'
   ├─ Check departure_time > NOW()
   └─ If invalid: ROLLBACK & Error 400
    ↓
3. CHECK SEAT AVAILABILITY
   ├─ if (schedule.available_seats < num_seats)
   └─ ROLLBACK & Error "Not enough seats"
    ↓
4. PROCESS PAYMENT
   ├─ Simulate payment gateway
   └─ If fails: ROLLBACK & Error "Payment failed"
    ↓
5. CREATE BOOKING
   ├─ INSERT INTO Bookings (...)
   ├─ TRIGGER EXECUTES: UPDATE Schedules SET available_seats = available_seats - num_seats
   └─ Within same transaction (atomic)
    ↓
6. VERIFY SEAT DEDUCTION
   ├─ SELECT Schedules WHERE id = schedule_id
   └─ Confirm available_seats decreased
    ↓
7. COMMIT or ROLLBACK
   ├─ All successful: COMMIT (persistent in database)
   └─ Any failed: ROLLBACK (undo everything)

GUARANTEES:
───────────────────────────────────────────────────────────────────────────────
✅ ATOMICITY: Booking + Seat Deduction = All or Nothing
   └─ Can't have booking without seat deduction
   └─ Can't have seat deduction without booking

✅ CONSISTENCY: Database Always Valid
   └─ available_seats never negative
   └─ available_seats never exceeds total_seats
   └─ Every booking has corresponding schedule

✅ ISOLATION: No Interference Between Bookings
   └─ SERIALIZABLE prevents dirty reads
   └─ SERIALIZABLE prevents phantom reads
   └─ Row locks prevent race conditions

✅ DURABILITY: Changes Persist
   └─ After COMMIT, changes are permanent
   └─ Even if server crashes, data safe
